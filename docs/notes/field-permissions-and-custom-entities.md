- 日期：2026-06-14

- 背景：产品侧提出两点需求——(1) 同一表上部分字段（如小分子的 `mol`/`smiles`）需单独授权才显示；

## 一、字段级权限

要求：行可查、敏感字段默认隐藏（返回空而非报错）、单独授权后可见。

三种机制（均已在本地 Postgres 实测）：

| 方案 | 机制 | 实测 | 成本 |
|---|---|---|---|
| 应用层裁剪 | 响应序列化时按用户权限删除字段键 | —（纯代码，必然可行） | **最低** |
| A 脱敏视图 + 会话变量 | 敏感列用 `CASE WHEN current_setting('app.fields_*')='on' THEN col END` 包视图；应用只授权读视图、禁读基表 | ✅ 未授权→NULL、行其余可见；事务内 `set_config` 授权→可见；出事务复位；直读基表被拒 | 中 |
| C 敏感列拆独立表 + RLS | 敏感列拆 `*_struct` 表，RLS 策略按会话变量放行 | ✅ 未授权 join→敏感列 NULL（RLS 隐藏行）；授权→可见 | 较高（多表/join） |

不适用：Postgres 原生列级 `GRANT(col)`——本应用单连接角色（`dms_app`）、且未授权列会**报错**而非隐藏。

**机制与现有架构同源**：A/C 都复用多租户 RLS 已在用的「每请求事务里 set 会话变量 + DB 强制」（现为 `app.current_tenant`，新增字段授权位即可）。

**成本分档建议（推荐）**：
- 绝大多数一般敏感字段 → **应用层裁剪**（最便宜）。
- 需要 DB 层兜底又不想拆表 → 方案 A。
- 仅极关键 / 有合规要求的少数字段（mol/smiles）→ 方案 C。
- 不要全表一刀切上重机制。

## 二、用户自定义实体（动态 Schema）

三种存储路线：

| 路线 | 评价 | 成本 |
|---|---|---|
| EAV（属性-值表） | 查询/索引/约束都痛 | 维护贵，不推荐 |
| **JSONB + Schema 注册表（推荐）** | 一张表 `data JSONB` 容纳不同类型；热点字段建 GIN / 表达式索引 | **低**（无需逐类型 DDL/迁移） |
| 动态建表（每类型一张真表） | 类型/索引最强 | 运行时 DDL 风险、RLS 逐表、表爆炸——运维贵 |

实测（JSONB 方案，临时验证已删）：
- 一张表同时存「小分子」「抗体」两种用户自定义类型及各自字段 ✅
- 按动态字段过滤（`data->>'formula'`）、JSONB containment（`data @> '{...}'`）✅
- 字段级脱敏 on JSONB：由 Schema 的 `sensitive` 标记**动态算出敏感键集合**并从 `data` 剔除，
  配合会话授权位 ✅（未授权返回 `{mw,formula}`，授权返回完整含 `mol/smiles`）

**成本建议**：
- JSONB 是最省的动态化路径（无逐类型迁移、单表、运维近零增量）。
- 但**需求明确前不建通用系统**：有第一个明确实体时先用一张普通表做实（最省最快）；
  待确有「用户自定义类型」强需求再泛化为 JSONB 注册表。避免提前抽象养复杂度。

---

## 三、统一设计（两需求一套元数据）

一张 **Schema 注册表**描述每个实体类型的字段，每字段带 `sensitive`/所需权限：
- 字段定义 → 驱动**动态实体**；
- 字段的 `sensitive` 标记 → 自动生成**字段级权限**并驱动脱敏。

即「用户定义实体 = 定义字段 + 标记哪些字段需单独授权」，一套元数据同时满足两需求。

## 四、成本原则（小公司）

- 能用 Postgres 解决就不加中间件（RLS/发件箱/JSONB 全在库内，免 Kafka/Redis/ES）。
- 能用应用层简单解决就不上 DB 重机制（字段裁剪优先）。
- 需求没定就先不建抽象，先把确定业务跑起来，贵能力按真实需求逐个开（对应复杂度分档）。

### 附：复现用关键 SQL（片段）

字段脱敏（会话变量，方案 A 核心）：
```sql
CASE WHEN current_setting('app.fields_structure', true)='on' THEN mol END AS mol
-- 请求事务内：SELECT set_config('app.fields_structure','on', true);
```
JSONB 按 Schema 动态剔除敏感键：
```sql
e.data - (SELECT coalesce(array_agg(f->>'key'), ARRAY[]::text[])
          FROM entity_type t, jsonb_array_elements(t.fields) f
          WHERE t.id = e.type_id AND (f->>'sensitive')::bool)
```
