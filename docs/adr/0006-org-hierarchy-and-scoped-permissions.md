# ADR-0006: 组织架构与作用域权限（对标 Benchling）

- 状态: 已接受
- 日期: 2026-06-14

## 背景

模板需要**通用**的组织/团队层级 + 资源级协作权限（用户明确要求做进模板）。参考 Benchling 的
设计：`租户 → 组织 → 团队 → 项目（数据/权限容器）`，权限**累积、取最宽松**。

## 决策

- **层级**：`租户(隔离边界) → 组织(平级) → 团队(平级)` + 资源协作者。组织/团队**不嵌套**
  （与 Benchling 一致，避免过度复杂）。
- **通用授予表 `role_grants`**：把某角色授予 `principal(用户|团队)`，作用域
  `scope(租户|组织|团队|资源)`。这一张表同时表达：组织/团队角色、项目协作者（资源作用域）。
- **累积解析**：用户有效权限 = 其本人 + 所属团队、在「租户级 + 目标作用域」上所有适用授予的
  权限**并集**。租户级沿用 `user_roles`（与登录 JWT 同源），作用域级用 `role_grants`，解析器
  合并两者。
- **资源权限不预烘焙进 JWT**：JWT 只带租户级权限；资源级权限按请求解析
  （`effective_permissions(user, scopes)`），与 Benchling「按资源解析」一致。
- 作为 **`orgs` feature**（依赖 `auth`），迁移 `migrations/orgs/0500`。

## 备选 / 刻意不做

- 嵌套组织树（部门/子部门）：更复杂、权限继承与查询成本高，暂不做（平级足够通用）。
- Benchling 的 access policy 可视化配置、schema 级策略、registry：太重，不做。
- 一个用户跨多租户：维持单租户归属（RLS 边界）。

## 影响

- 通用、与 Benchling 思路一致；新业务实体可直接复用 `role_grants` 做资源协作者。
- 资源权限按请求解析（JWT 保持短小）；解析逻辑清晰、可单测。
- 复杂度可控：`orgs` 为可选档，简单项目不启用即无此负担。
